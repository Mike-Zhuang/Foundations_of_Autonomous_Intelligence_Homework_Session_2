## 文件结构

- `README.md`：项目说明与协作规范。
- `install.md`：开发环境安装说明。
- `git_use.md`：Git 基础使用教程。

  - `Birdnet声纹识别.md`：BirdNET 声纹识别资源下载说明。

  - `校园生物多样性观测记录表.xlsx`：观测记录表模板。

  - `panorama/`：全景类图片（含 `origional_heic/` 原始 HEIC 文件）。
  - `plants/`：植物类图片（含 `origional_heic/` 原始 HEIC 文件）。
- `yolo_recognition/`：YOLO 相关脚本、模型与操作文档。
  - `yolo_process.py`：视频检测与结果输出脚本。
  - `yolo_read.py`：检测结果读取脚本。
  - `yolo26s.pt`：模型权重文件。
  - `YOLO人车交通识别操作说明.md`：操作文档。
- `doc_images/`：文档配图资源目录（含 README、install、git_use 与 YOLO 手册图片）。
- `LICENSE`：许可条款。

## 图片说明

- 本次整理中，图片按内容分为两类：全景图片与植物图片，分别放在 `panorama` 和 `plants` 子目录。
- 实际操作中，若无兼容性限制，优先使用各子目录中的 `origional_heic` 目录内 HEIC 格式图片（通常在相近视觉质量下体积更优、保留信息更多）。
- 需要跨平台或网页展示时，可使用已经转换好的 JPG 格式。

## 协作规范

1. 禁止任何人直接向 `main` 分支提交（包括 `push`、`merge`、`rebase` 后推送）。
2. 所有改动必须在个人功能分支完成，分支命名建议：`feature/*`、`fix/*`、`docs/*`。
3. 所有改动必须通过 Pull Request 合并，且 PR 必须包含变更说明。
4. 未经 Review Approve，不允许合并。
5. 严禁“先合并后补文档/补测试/补说明”。文档与必要验证必须随 PR 一并提交。
6. 完成改动后，建议在微信同步说明。

## 提交协议

1. `main` 仅允许通过 PR 合并，禁止直接推送（项目说明文档除外）。
2. PR 必须由非提交者审批通过后方可合并。
3. PR 必须通过必要检查（至少包含基本可读性与内容完整性检查）后方可合并。
4. 紧急变更在微信群内讨论并取得同意后，可作为例外处理。
5. 若出现误推 `main`，必须立即：
  - 在协作渠道（微信）公告
  - 回滚至最近稳定提交
